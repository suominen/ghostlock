GhostLock (CVE-2026-43499) is a stack use-after-free in the Linux kernel's
real-time mutex (rtmutex) priority-inheritance code, reached through the
futex requeue-PI path. remove_waiter() in kernel/locking/rtmutex.c clears
pi_blocked_on on `current` instead of on the waiter's own task; when the
requeue-PI rollback runs it on behalf of a DIFFERENT task (a PI deadlock
cycle across three futexes returns -EDEADLK and takes the buggy rollback
path), it leaves a dangling pointer into freed kernel-stack memory — a
use-after-free the PoC turns into a near-arbitrary write, local root, and
container escape. The trigger is ordinary futex(2), so any unprivileged
local task (including one inside an unprivileged container) can reach it;
there is no privilege, device, capability, or user-namespace gate, and it
is architecture-independent. The canonical fix is 3bfdc63936dd (rtmutex:
Use waiter::task instead of current in remove_waiter()), first released in
v7.1. The bug was introduced by 8161239a8bcc in v2.6.39 (2011), so
essentially every kernel from 2.6.39 through 7.0 is in-window.
